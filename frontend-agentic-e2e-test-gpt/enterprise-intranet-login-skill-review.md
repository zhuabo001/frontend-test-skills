# run-playwright-mcp-testcase 企业内网登录处理审查

## 问题

分析 `skills-bundle/run-playwright-mcp-testcase/` 这个 skill 中关于企业内网登录的处理是否合理；如果不合理，给出理由。

## 结论

作为一般性的 E2E 执行流程，该 skill 的认证处理思路基本正确；但作为企业内网登录规范，还不够严格，不能直接视为可安全落地的控制措施。

合理的部分包括：

- 默认要求用户在可见浏览器中手动完成企业 SSO/MFA。
- 禁止用户在对话中提交账号、密码、验证码等凭据。
- 认证失败或认证不可用时，将结果归类为 `blocked`，而不是产品功能失败。
- 自动凭据登录只允许在明确授权、非生产测试账号、且具备 secret-safe 能力的情况下使用。

## 主要问题

### 1. `storage_state` 的保护标准不足

当前要求是 `storage_state` 必须放在仓库外，或放在被忽略的路径中。这只能降低误提交风险，不能构成安全控制。

`storage_state` 往往包含 cookie、token、会话标识，本质上相当于 bearer credential。即使 `.gitignore` 生效，也无法防止本机其他进程、同步盘、备份软件、日志采集或误传输读取该文件。

更合理的要求应包括：

- 默认不持久化企业内网登录态。
- 如必须持久化，应使用受控临时目录或系统密钥存储。
- 校验真实路径和符号链接，避免绕过路径约束。
- 设置最小文件权限。
- 明确到期、清理和撤销策略。

### 2. 复用既有登录会话缺少身份和租户边界

当前逻辑允许复用已有 MCP 浏览器登录态，只要能访问受保护目标即可。但“能访问页面”不足以证明该会话适合测试。

缺失的校验包括：

- 是否为用户明确授权用于本次测试的账号。
- 是否为专用测试账号，而非个人办公账号。
- 是否属于正确企业租户。
- 是否具备最小权限。
- 是否会在测试过程中触发有副作用的内网操作。

这会带来一个实际风险：测试可能在用户的个人高权限企业会话中运行，导致数据读取、数据修改、审计记录或权限边界都不可控。

### 3. 缺少 SSO 跳转域名和签发方白名单

企业 SSO 的风险不只在输入密码，还在重定向链路。当前流程没有明确要求校验：

- 业务入口域名。
- IdP 域名。
- 回调域名。
- 租户标识。
- token issuer / audience。

遇到异常登录域、未知回调、跨租户账号选择页或非预期 IdP 时，应暂停并要求用户确认，而不是继续执行。

### 4. 敏感信息保护偏声明式，缺少可执行控制

skill 要求不要写入凭据、token、cookie、Authorization header 等敏感信息，这是正确方向。但现有报告校验主要按 JSON 字段名识别敏感字段，不能可靠发现以下泄露：

- `diagnostic` 字符串中的 token 或个人信息。
- 网络错误文本中的 Authorization header、cookie、URL query token。
- 控制台日志中的内网数据。
- 截图中的账号名、工号、邮箱、部门、客户数据或 token。
- artifact 文件中的响应正文或 HAR 内容。

因此，“不记录敏感信息”目前更像流程要求，而不是强约束。企业内网场景需要更强的默认脱敏和证据最小化策略。

### 5. 缺少认证会话收尾策略

流程说明了如何复用、手动登录和隔离会话，但没有明确测试结束后的处理方式。

应补充：

- 关闭专用 browser context。
- 删除临时 `storage_state`。
- 不继续复用本次临时登录态。
- 必要时提示用户撤销测试会话或退出登录。
- 明确个人既有浏览器会话不可由 agent 擅自清理，但需要提示残留风险。

否则企业内网登录态可能残留，并被后续无关任务误用。

## 建议默认策略

更稳妥的企业内网登录默认策略应是：

- 仅允许用户在专用、可见、隔离的浏览器上下文中手动登录。
- 登录前确认业务域、IdP 域、回调域和租户白名单。
- 禁止复用未明确授权的既有个人会话。
- 默认不保存 `storage_state`。
- 网络证据采用白名单采集，默认不保存请求头、响应体、cookie、token 或完整 HAR。
- 截图前尽量避开或遮蔽账号、工号、邮箱、客户数据等敏感信息。
- 测试结束后清理临时认证状态，并在报告中记录清理结果。

## 总体判断

该 skill 对企业登录的方向是正确的：人工完成 SSO/MFA、禁止聊天传凭据、认证问题不算产品失败。但它目前仍停留在“流程约束”为主，缺少足够的技术防护、边界校验、证据脱敏和会话清理。

如果用于普通测试环境，可以接受；如果用于真实企业内网或生产级 SSO 环境，需要先补强上述控制。
